# Проект Нимбус: Техническая документация

## Архитектура системы

Нимбус построен на микросервисной архитектуре. Всего 14 микросервисов, объединённых через внутреннюю шину сообщений «Бриз» (разработана в Zephyra, основана на протоколе ZephyrMQ).

### Основные микросервисы

1. **SensorHub** — сбор данных с датчиков. Поддерживает протоколы ZigBee, LoRa и проприетарный ZephyrLink. Частота опроса — 1 раз в 3 секунды для критических датчиков (температура, CO2) и 1 раз в 30 секунд для вторичных (освещённость, шум).

2. **ClimateEngine** — алгоритм принятия решений по управлению климатом. Использует нейросеть «Cirrus-Net» (разработана Zephyra, 12 миллионов параметров). Cirrus-Net обучена на данных из 2300 теплиц-партнёров.

3. **DroneOrchestrator** — управление парком дронов Пчёлка-7. Максимум 200 дронов на одну теплицу. Планирование маршрутов через алгоритм «Вихрь» (модификация A* с учётом потоков воздуха в теплице).

4. **AlertSystem** — система оповещений. Три уровня критичности: «Бриз» (информационный), «Шторм» (предупреждение), «Торнадо» (критический). При уровне «Торнадо» автоматически звонит дежурному агроному.

5. **HarvestPredictor** — прогнозирование урожайности на основе алгоритма ВетроПрогноз-3.

6. **DataLake** — хранилище исторических данных. Использует формат ZephyrParquet (расширение Apache Parquet с дополнительными метаданными о теплицах).

### База данных

Нимбус использует три базы данных:
- **CrystalDB** — проприетарная time-series база для данных с датчиков (до 50 миллионов записей в день)
- **PostgreSQL** — для пользовательских данных, настроек, конфигурации теплиц
- **Redis** — кэширование текущих показаний и статусов дронов

### API

Внешний API Нимбуса доступен через REST и gRPC. Документация API находится по адресу api.zephyra.tech/docs (внутренний сервис). Аутентификация — через ZephyraToken (JWT с дополнительным полем greenhouse_id).

Лимиты API:
- Free план: 1000 запросов в час
- Pro план: 50000 запросов в час
- Enterprise: без лимитов

## Процесс разработки

### Ветвление

Используется модель GitZephyr Flow:
- `main` — продакшн код
- `storm/{ticket-id}` — фича-ветки (вместо feature/)
- `frost/{ticket-id}` — багфиксы (вместо fix/)
- `aurora/release-X.Y` — релизные ветки

### Code Review

Каждый Pull Request проходит проверку минимум двух ревьюеров. Для критических сервисов (SensorHub, ClimateEngine) требуется одобрение Z6+ сотрудника. Среднее время ревью — 4 часа. Максимальный размер PR — 400 строк (правило «Порыв ветра» — если больше, нужно разбить).

### Тестирование

- Unit-тесты: покрытие не менее 80%
- Интеграционные тесты: запускаются в среде «Симулятор Теплицы» (эмулирует 100 датчиков)
- E2E тесты: раз в неделю на пилотной теплице «Альфа-Парник» в пригороде Кристаллограда
- Нагрузочное тестирование: инструмент «Ураган» (внутренняя разработка), симулирует нагрузку до 10000 теплиц

## Инциденты

### Инцидент «Ледяной шторм» (март 2024)

ClimateEngine из-за бага в Cirrus-Net v2.1 интерпретировал показания влажности 85% как 8.5%, что привело к аварийному включению полива в 47 теплицах одновременно. Время обнаружения — 23 минуты. Время устранения — 1 час 40 минут. Убыток — 2.3 миллиона зефиров. После инцидента введено правило «Валидация Абсурда» — ClimateEngine проверяет, что изменение показателя не превышает 30% за 1 минуту.

### Инцидент «Потерянный рой» (август 2024)

12 дронов Пчёлка-7 из-за ошибки в DroneOrchestrator v3.4 одновременно попытались приземлиться на одну станцию «Улей». 8 дронов столкнулись и вышли из строя. Причина — race condition в алгоритме «Вихрь». Исправлено добавлением распределённого мьютекса «Штиль».
